01 · Background

왜 TripMate를 기획했는가

솔로 여행 수요는 빠르게 늘었지만, 동행을 구하는 방식은 여전히 카페 글쓰기·SNS 모집·지인 네트워크에 머물러 있습니다. 사용자 인터뷰에서 반복적으로 나온 세 가지 문제에서 출발했습니다.

01
Problem · 탐색
동행 탐색의
비효율
카페 글쓰기·SNS 모집·지인 추천에 의존하는 방식은 시간이 많이 걸리고 성공률이 낮습니다. 원하는 조건으로 직접 필터링할 도구가 없었습니다.
카페에 글 올려도 연락이 없거나, 스타일이 너무 달라서 포기했어요.
02
Problem · 신뢰
신뢰할 수 없는
동행자
온라인에서 만난 낯선 사람과 함께 여행하는 것에 대한 불안감이 큽니다. 프로필 검증 방법도, 성향을 미리 가늠할 수단도 부족합니다.
프로필 사진 한 장만 보고 연락하기엔 너무 무섭고 불안해요.
03
Problem · 궁합
여행 스타일
불일치
같은 목적지라도 빠른 관광 ↔ 느린 힐링, 아침형 ↔ 저녁형 등 스타일 차이로 동행 갈등이 자주 발생합니다. 사전에 스타일을 맞출 장치가 없었습니다.
일정표도 다르고 일어나는 시간도 달라서 결국 따로 다녔어요.

03 · Target Persona

누구를 위한
서비스인가

TripMate의 핵심 타겟은 솔로 여행을 즐기지만 믿을 만한 동행을 원하는 20~30대입니다. 두 가지 페르소나를 중심으로 서비스 경험을 설계했습니다.

GOAL
비슷한 취향과 일정의 동행자와 함께 안전하고 즐거운 첫 솔로 여행을 경험하고 싶다.

GOAL
여행 취향과 페이스가 맞는 동행자와 더 풍부하고 새로운 여행 경험을 쌓고 싶다.

06 · Matching Model

매칭률 산출 모델

Express 백엔드(server/src/lib/score.ts)에 구현되어 동작 중입니다. 일정 겹침(40%) + 스타일 코사인 유사도(40%) + 신뢰도(20%)를 가중 평균해 0–100 점수를 계산하고, 상위 10명을 정렬해 반환합니다.

+ · State & Data Architecture

상태 관리와 데이터 흐름

실서버 없이도 전체 사용자 경험을 검증할 수 있도록 Mock-first 구조 위에 4계층으로 데이터를 분리했습니다. UI → Context → Service → Data 한 방향 흐름.

01
UI Layer
35+개 화면 · Expo Router 라우팅. Context를 직접 구독.
02
Context Layer
도메인별 5개 Context. AsyncStorage에 자동 영속화.
03
Service Layer
매칭/채팅/커뮤니티 서비스. mock 또는 실 API 호출.
04
Data Source
Express + Prisma + Postgres 실서버에 연결됨. 환경변수로 mock ↔ real 전환.
Domain Context
5개
AuthContext
user · signIn / signOut · 토큰 영속화
ProfileContext
아바타 · 바이오 · MBTI · 스타일 · 찜한 글
PersonalityContext
7차원 여행 성향 응답 저장
TripsContext
upcoming · history · recent 트립 관리
VerificationContext
신뢰도 점수(0–100) · 색상 밴드
Service Layer
4개
matchService
findMates / getRecommended / likeMate
chatService
데모 룸(c1·c2·c3) + 동적 룸, 동행 카드
communityService
mate · tips · review 카테고리 피드
api.ts

09 · Tech Stack

기술 스택

React Native + Expo로 iOS/Android/Web을 한 코드베이스로 구현. 백엔드는 Express + Prisma + Postgres로 V1을 직접 구현해 실 데이터까지 연결했습니다. 실시간 채팅은 V1에서 HTTP 폴링으로 시작하고, WebSocket / 외부 AI 서버는 V2 확장 과제로 분리했습니다.
